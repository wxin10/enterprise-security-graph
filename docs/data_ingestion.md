# 数据入图与清洗链路设计

## 1. 设计目标
为了支撑“日志驱动型”的安全图谱构建，本系统设计了一套轻量、可扩展的数据接入链路。
目标是通过脚本化的流水线，清晰展示从“非结构化/半结构化原始日志”转化为“基于 Neo4j 的高质量图谱数据”的全过程，便于毕业设计论文中对“数据治理与知识抽取”章节进行阐述。

## 2. 原始数据格式与来源
当前系统的基础日志来源主要模拟企业环境中的几类常见日志：

1. **认证与访问日志 (Auth/Access Logs)**：格式为 JSON 或泛结构化文本，记录用户登录登出、IP 来源、访问目标资产和端口协议等基础流量属性。
2. **安全行为日志 (Security Event Logs)**：来自防病毒软件、IDS/IPS 系统生成的原始安全事件，通常记录事件类型、严重等级、源与目的信息。
3. **运维与治理操作日志 (Syslog/Audit)**：用于展示封禁与审批执行情况。

**原始字段形态示例**：
`{"timestamp": "2026-04-01T10:00:00", "src_ip": "192.168.1.100", "user": "admin", "action": "login_failed", "dst_host": "DB_SERVER"}`

## 3. 核心入图链路流程

系统的入图链路分为三个主要步骤，代码实现在 `scripts/` 目录下：

### 阶段一：日志监控与初步清洗 (`log_watcher.py` / `clean_logs.py`)
- **动作**：通过轮询 `data/incoming/` 目录发现新产生的日志切片。
- **目标**：完成 JSON 等格式的解析、非法字符过滤、时间戳统一标准化。

### 阶段二：安全实体与关系抽取 (`extract_entities.py`)
- **动作**：将展平的日志记录分解为图谱所需的点、边集合。
- **机制**：
  - 提取 `user` 字段 -> 生成 `User` 节点核心属性。
  - 提取 `src_ip` 字段 -> 生成 `IP` 节点核心属性。
  - 构建过程上下文 -> 生成 `Session` 节点，并计算此次行为的初始风险分 (Risk Score)。
  - 解析 `action` -> 形成 `Event` 节点。
- **落盘**：为了降低数据库直接高频写入压力，抽取结果会先保存至 `data/processed/` 下的系列标准化 CSV 文件（如 `users.csv`, `rel_user_initiates_session.csv`）。

### 阶段三：图数据库批量入站 (`import_to_neo4j.py`)
- **动作**：利用 Neo4j Python 驱动（代替繁重的手动 LOAD CSV），按批次将编排好的节点和关系提交到图数据库。
- **机制**：使用 `MERGE` 语句防止节点与关系重复构建，并通过预置的 schema 定义（`init_schema.cypher`）保证唯一性约束。

## 4. 字段映射关系（关键映射表）

为说明“哪些字段承担检测能力”，以下是部分核心字段向图实体的映射与作用划分：

| 原始日志字段 | 映射目标实体/关系 | 转换后图节点核心字段 | 业务作用与说明 |
| --- | --- | --- | --- |
| `src_ip` | `IP` 节点 | `ip_address` | **封禁核心主键**，系统所有的自动化防火墙阻断、恢复操作均基于该字段定位。 |
| `user` | `User` 节点 | `username`, `department` | 辅助进行账号级别的权限核验和异常漫游检测。 |
| `action` | `Event` 节点 | `event_type`, `event_level` | **风险识别输入**。安全规则匹配最基本的研判依据。 |
| `timestamp` / `duration` | `Session` 节点 | `start_time`, `duration_seconds` | 用于构建时间窗攻击序列，是识别“短时高频访问”与构建“攻击链”的基础时序约束。 |
| (规则计算产生) | `Alert` 节点 | `severity`, `score` | 是生成 `BlockAction` (封禁动作) 的直接触发条件。 |

## 5. 建模与分析价值 (论文适用)

此套导入链路将非结构化日志升级为图谱结构，为后续的恶意行为检测带来了两个维度上的突破：

1. **时序与空间关联的可解释性**：常规日志审计系统只能分析单条“登录失败”行为。但在本系统图谱中，通过 `Session` 作为中枢，可直观查询 `IP` -> `Session` -> `Event(LOGIN_FAIL)` -> `Host`，极大提升了多步横向移动与 APT 攻击的追溯能力。
2. **快速闭环联动**：从 `Alert` 发起到 `BlockAction` 执行落库，同样融入图中（`DISPOSES` / `TARGETS_IP`），可以直接量化评估阻断策略的覆盖率以及被阻断实体的后续活跃度。当检测到某一高危主机被访问时，可以直接依托图路径提取关联的存活 `Session` 并对关联 `IP` 进行自动阻断与审批。
