## 2. 最高优先级规则：先检查现状，再修改

你不能假设项目是初始状态。
你不能假设项目没有被改过。
你不能假设某个文件一定存在或一定不存在。

在开始任何修改前，必须先根据当前真实代码状态进行检查，然后再决定方案。

### 每次开始工作时的固定流程

#### 第一步：先读取进度文件
开始任何分析或编码前，必须先读取：

- `CODEX_PROGRESS.md`
- `CODEX_NEXT.md`

读取后先理解：

- 上一批已经改了哪些文件
- 哪些内容已成功落地
- 哪些内容待验证
- 当前下一批应该处理哪些文件
- 当前是否处于“恢复中断”状态

#### 第二步：检查当前工作区状态
必须运行或等价检查：

- `git status --short`

如果工作区不是干净状态，必须先判断：

- 哪些变更是本轮要接着做的
- 哪些变更是上轮中断遗留的
- 哪些文件已经改成功
- 哪些文件还没真正写进去

#### 第三步：只核查当前批次相关文件
只读取当前批次要处理的文件，不要无意义扫全仓库。
但如果用户明确要求“整体现状分析”，则应先做仓库级检查。

#### 第四步：输出现状结论
在开始改代码前，必须明确说明：

1. 当前工作区是否干净
2. 当前批次准备改哪些文件
3. 这些文件为什么要改
4. 这次不会改哪些文件
5. 当前是否存在中断恢复情形

#### 第五步：再开始编码
不允许跳过以上步骤直接开改。

## 3. 角色模型（非常重要，不能理解错）

本系统是**内部运维 / 安全人员使用的系统**，不是外部访客系统。

### 错误理解（禁止）
- 普通用户 = 访客
- 普通用户 = 只能看个人信息
- 普通用户 = 外部业务用户

### 正确理解（必须采用）
- **普通用户** = 一线运维 / 值班安全分析员
- **管理员** = 高权限运维负责人 / 平台管理员

### 角色设计原则
管理员和普通用户都可以访问**核心安全业务页面**，区别不在“能不能看”，而在：

- 是否能管理系统用户
- 是否能管理规则
- 是否能审批
- 是否能执行最终高风险操作
- 是否能修改系统关键配置
- 是否能查看完整审计日志

### 普通用户应该具备的能力
普通用户不是访客，必须具备：

- 查看系统总览 / 工作台
- 查看检测结果
- 查看恶意行为告警
- 查看图谱分析 / 攻击链分析
- 查看详情页
- 查看个人中心
- 对告警进行查看、筛选、分析、标记
- 填写处置意见
- 发起封禁申请 / 处置申请
- 查看自己的处理记录

### 普通用户不能做的事
普通用户不能：

- 管理系统用户
- 修改角色权限
- 修改识别规则 / 封禁规则
- 执行最终封禁 / 解封
- 修改系统关键配置
- 查看完整审计日志

### 管理员应该具备的能力
管理员除拥有公共业务访问能力外，还应具备：

- 用户管理
- 角色权限管理
- 规则管理
- 封禁审批 / 最终封禁 / 解封
- 审计日志查看
- 系统关键配置修改

## 4. 页面与菜单设计原则

### 公共业务页面（管理员和普通用户都能访问）
以下页面默认属于公共业务页面：

- 系统总览 / 工作台
- 检测结果页
- 告警中心 / 恶意行为记录页
- 图谱分析 / 攻击链分析页
- 详情页
- 个人中心页

### 普通用户菜单建议
普通用户菜单应以业务处理为主，至少包括：

- 工作台
- 检测结果
- 告警中心
- 图谱分析
- 封禁申请 / 处置申请
- 我的处理记录
- 个人中心

### 管理员菜单建议
管理员菜单在公共菜单基础上增加管理模块，至少包括：

- 工作台
- 检测结果
- 告警中心
- 图谱分析
- 封禁审批 / 封禁管理
- 用户管理
- 规则管理
- 审计日志
- 个人中心

### 路由权限原则
- 管理员可以访问管理员专属页面
- 普通用户不能访问管理员专属页面
- 普通用户手动输入管理员路由时，应跳转到无权限页或普通用户首页
- 不要把核心业务页全部只给管理员

## 5. 白色主题改造原则

本系统最终页面风格要求：

- 整体主色调为白色
- 当前灰色背景要统一过渡到白色背景
- 风格简洁、干净、适合毕业设计展示
- 偏后台管理系统风格
- 不要花哨，不要引入不必要的炫技动画

### 改造原则
- 优先复用现有结构
- 优先统一全局样式入口
- 不要每个页面各自为战
- 颜色、卡片、按钮、输入框、导航栏风格要统一
- 若局部页面需要修补，先走“全局统一 + 局部微调”路径

## 6. 文件修改策略

### 核心原则
- 优先增量修改
- 不要无意义重构
- 不要重复造轮子
- 不要推翻现有结构
- 不要直接覆盖用户已有实现

### 批次策略
为了降低中途中断、断流和状态不清的问题，默认采用小批次执行：

#### 默认规则
- 默认每批最多处理 **2 个文件**
- 如果上一批中途断流，则下一批自动降级为 **1 个文件**
- 只有当多个文件强相关、且用户明确允许时，才可以处理 3 个文件
- 不允许在同一批里同时改“路由 + 布局 + 大量页面 + 样式 + 规则页”这种大跨度内容

### 每批改动后必须立刻停止
每完成一批，必须：

1. 输出本批实际改动了哪些文件
2. 输出 `git status --short`
3. 更新 `CODEX_PROGRESS.md`
4. 更新 `CODEX_NEXT.md`
5. 明确说明下一批建议处理什么

未做完这些，不得直接跳下一批。

## 7. 中断恢复规则（非常重要）

如果上一轮发生下列情况之一：

- 网络断流
- stream disconnected
- 输出被截断
- 没有看到 git 状态
- 没有看到完整文件内容
- 没有看到“已修改哪些文件”的结果

则必须先进入**恢复模式**，而不是直接继续写代码。

### 恢复模式固定流程
1. 先读取 `CODEX_PROGRESS.md`
2. 再运行或等价检查 `git status --short`
3. 核对上一批声称修改的文件是否真的存在于工作区
4. 判断哪些文件已成功落地，哪些只是计划未落地
5. 用一句话先告诉用户：
   - 已落地哪些文件
   - 未落地哪些文件
6. 然后只从“未完成的最小一批”继续

### 恢复模式下的禁止事项
- 不要从头重做整个任务
- 不要重新覆盖已经成功落地的文件
- 不要凭记忆判断“应该已经改过”
- 不要直接跳过核对阶段

- 本批准备处理哪些文件

2. 当前 `git status --short`

### 中文要求
- 所有说明用中文
- 所有新增代码注释优先用中文
- 复杂逻辑必须写清楚注释

- 未经说明直接改依赖版本

如果确实需要做上述操作，必须先明确说明原因和影响，再等用户确认。
