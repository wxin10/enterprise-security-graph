# 当前进度说明

## 1. 当前已完成的核心能力

当前项目已完成并落地的核心能力如下：

1. 已形成完整工程目录，而非只有建模文档。
2. 已实现 `backend/` Flask 后端，包含图谱、监控、告警、封禁等接口。
3. 已实现 `frontend/` Vue3 前端，包含仪表盘、监控中心、告警与封禁管理等页面。
4. 已实现 `scripts/` 目录下的日志监测、分类、清洗、抽取、导入、识别与行为映射脚本。
5. 已实现 Neo4j 图谱导入与攻击关系展示所需的基础数据链路。
6. 已实现自动封禁、放行恢复、执行状态校验与动作历史审计。
7. 已完成 Windows 防火墙主机级自动封禁测试链路验证。

## 2. 当前版本采用的自动封禁策略

当前版本采用“先识别高危行为，再触发动作”的策略，重点验证自动处置闭环，而不是只生成中间数据。

现阶段自动封禁策略特点如下：

1. 日志统一从 `data/incoming/` 或 `data/incoming/unified/` 接入。
2. 系统会对输入日志做轻量分类、标准化、行为识别和风险评估。
3. 对高危攻击类型优先触发自动封禁，当前已覆盖的主线包括：
   - SQL 注入
   - 命令注入
   - 暴力破解
   - 横向移动
4. 默认按风险分数阈值控制自动封禁，当前代码中主机级自动封禁最低风险分默认值为 `85`。
5. 对暴力破解场景，除风险分外还结合事件次数进行触发。
6. 封禁执行层当前支持模拟执行、应用层阻断以及 Windows 防火墙主机级阻断等方式，其中本地演示重点验证 Windows 防火墙链路。

## 3. 当前版本已验证通过的链路

当前已验证通过、可直接用于演示的主线如下：

1. 攻击前目标服务可正常访问。
2. 将高危测试日志导入统一入口目录。
3. 系统自动识别攻击行为并生成封禁动作。
4. 后端记录封禁状态、动作历史和执行结果。
5. Windows 主机生成对应防火墙阻断规则。
6. 攻击机再次访问目标服务时失败。
7. 管理端执行放行后，阻断规则移除，访问恢复。

## 4. 当前仍可继续优化但不影响可用性的部分

以下内容仍可继续优化，但不影响当前版本作为毕业设计工程原型进行展示和评估：

1. 攻击识别规则覆盖面还可以继续扩展。
2. 自动封禁策略还可以进一步细化为更多分级处置策略。
3. 监控链路当前以本地目录轮询和脚本驱动为主，后续可升级为更稳定的消息化或服务化方案。
4. 前端展示与统计报表还可以继续增强，但现有页面已能支撑基础演示。
5. 对 Linux 或更多网络设备的真实联动处置仍可继续扩展。
