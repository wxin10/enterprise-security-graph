# 基于 Neo4j 的企业网络恶意行为识别与封禁系统

## 项目概述

本项目是毕业设计《基于 Neo4j 的企业网络恶意行为识别与封禁系统的设计与实现》的工程实现仓库。

当前仓库已经不再是“第一阶段仅做建模文档”的状态，而是包含可运行的后端、前端、日志处理脚本、Neo4j 导入脚本以及自动封禁相关能力，重点验证“日志输入 -> 攻击识别 -> 自动封禁 -> 阻断验证 -> 放行恢复”的闭环。

当前版本的重点不是只产出中间 CSV 或图谱数据，而是强调动作导向的安全处置链路：识别高危攻击后，系统能够自动形成告警、生成封禁动作，并在演示环境中完成主机级阻断与恢复。

详细进度说明见：[docs/current_status.md](docs/current_status.md)。

## 当前已具备的核心能力

1. 日志接入与分类
   - 支持 `data/incoming/` 下多目录输入。
   - 支持统一入口 `data/incoming/unified/` 的自动识别与路由。
2. 攻击识别与图谱入库
   - 支持日志清洗、实体抽取、Neo4j 导入、行为识别与告警映射。
   - 可将用户、主机、IP、会话、事件、告警、封禁动作写入图数据库并进行展示。
3. 自动封禁与审计
   - 后端提供封禁列表、详情、校验、放行、重新封禁等接口。
   - 封禁动作保留当前状态与历史动作，便于审计和答辩展示。
4. Windows 防火墙主机级阻断
   - 当前版本已具备 Windows 主机环境下的自动封禁测试链路。
   - 支持生成项目专用 Windows 防火墙规则并校验规则是否生效。
5. 放行恢复
   - 支持对已封禁目标执行放行。
   - 放行后可校验规则移除，并恢复目标访问。
6. 前端展示
   - 已具备仪表盘、监控中心、告警、封禁管理、攻击关系展示等页面基础能力。

## 当前版本的核心主线

当前仓库围绕以下主线组织，而不是只停留在建模文档阶段：

1. 日志输入：测试日志放入 `data/incoming/` 或 `data/incoming/unified/`
2. 攻击识别：脚本完成分类、清洗、行为识别、告警映射与 Neo4j 入库
3. 自动封禁：识别到高危攻击后生成封禁动作并记录审计信息
4. Windows 防火墙主机级阻断：在 Windows 演示环境中下发并校验阻断规则
5. 放行恢复：执行解除封禁后恢复访问，并保留完整动作历史

## 当前演示主线

当前版本适合按以下顺序进行答辩或演示：

1. 攻击前目标服务可正常访问
2. 导入高危攻击日志或统一入口测试日志
3. 系统自动识别攻击并触发封禁
4. Windows 防火墙生成对应阻断规则
5. 攻击机再次访问目标服务失败
6. 在管理端执行放行后，目标访问恢复

## 运行与环境提醒

1. Neo4j 建议使用 5.x，Python 侧使用官方 `neo4j` 驱动。
2. 前后端联调时需注意跨域配置。
3. Windows 防火墙主机级自动封禁链路需要在 Windows 环境下运行，并具备相应权限。
4. 若未启用真实防火墙执行模式，系统仍可使用模拟执行或应用层阻断方式进行演示。
